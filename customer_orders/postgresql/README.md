# Customer Orders Schema — PostgreSQL 17 Migration

This directory contains the PostgreSQL 17 port of the Oracle 19c **Customer Orders (CO)** sample schema.

## Quick Start

```bash
# Install (creates schema 'co' by default)
psql -d your_database -f co_install.sql

# Install into a custom schema name
psql -d your_database -v co_schema=my_co -f co_install.sql

# Validate the migration
psql -d your_database -v co_schema=co -f co_validate.sql

# Uninstall
psql -d your_database -f co_uninstall.sql
```

## Key Conversion Rules Applied

### Data Types

| Oracle | PostgreSQL | Notes |
|--------|-----------|-------|
| `VARCHAR2(n CHAR)` | `VARCHAR(n)` | Character semantics preserved |
| `NUMBER(p,s)` | `NUMERIC(p,s)` | Exact precision match |
| `BLOB` (JSON data) | `JSONB` | `product_details` column — enables native JSON queries |
| `BLOB` (binary data) | `BYTEA` | `logo`, `product_image` columns |
| `DATE` | `TIMESTAMP` | Oracle `DATE` includes a time component |
| `INTEGER` | `INTEGER` | No change needed |

### Identity Columns

| Oracle | PostgreSQL |
|--------|-----------|
| `GENERATED BY DEFAULT ON NULL AS IDENTITY` | `GENERATED BY DEFAULT AS IDENTITY` |
| `ALTER TABLE ... MODIFY ... START WITH LIMIT VALUE` | `SELECT setval(pg_get_serial_sequence(...), (SELECT MAX(...)))` |

**Behavioral difference:** Oracle's `ON NULL` clause auto-generates a value when an explicit `NULL` is inserted. PostgreSQL does **not** support this — an explicit `NULL` insert will fail. This does not affect the data load (which always supplies explicit IDs), but callers should be aware.

### View Translations

| Oracle Construct | PostgreSQL Equivalent | Used In |
|-----------------|----------------------|---------|
| `LISTAGG(col, ', ' ON OVERFLOW TRUNCATE '...' WITH COUNT) WITHIN GROUP (ORDER BY ...)` | `STRING_AGG(col, ', ' ORDER BY ...)` | `customer_order_products` |
| `GROUPING_ID(a, b)` | `GROUPING(a) * 2 + GROUPING(b)` | `store_orders` |
| `JSON_TABLE(blob, '$.reviews[*]' COLUMNS (...))` | `LATERAL jsonb_array_elements(col->'reviews')` with `->>'field'` access | `product_reviews` |
| `NVL(a, b)` | `COALESCE(a, b)` | `store_orders` |

### Constraints

| Oracle | PostgreSQL | Notes |
|--------|-----------|-------|
| `CHECK (product_details IS JSON)` | *(removed)* | `JSONB` type inherently validates JSON |
| All PK, FK, UNIQUE, CHECK | Preserved | Same names, same semantics |

### Scripting / psql

| Oracle SQL*Plus | PostgreSQL psql |
|----------------|----------------|
| `SPOOL file.log` | `\o file.log` |
| `@@co_create.sql` | `\i co_create.sql` |
| `ACCEPT var PROMPT '...'` | *(removed — use `-v` flag)* |
| `WHENEVER SQLERROR EXIT` | `\set ON_ERROR_STOP on` |
| `SET ECHO OFF` / `SET FEEDBACK OFF` | *(removed — psql defaults)* |
| `Prompt message` | `\echo 'message'` |
| `ALTER SESSION SET CURRENT_SCHEMA=CO` | `SET search_path TO co, public` |
| `CREATE USER co ... / DROP USER co CASCADE` | `CREATE SCHEMA co / DROP SCHEMA co CASCADE` |
| `BEGIN ... END; /` (PL/SQL anonymous blocks wrapping INSERTs) | Plain SQL `INSERT` statements |

## Known Behavioral Differences

### 1. STRING_AGG vs LISTAGG Overflow

Oracle's `LISTAGG ... ON OVERFLOW TRUNCATE '...' WITH COUNT` truncates the aggregated string if it exceeds the return type limit and appends a count of omitted values. PostgreSQL's `STRING_AGG` returns the **full** concatenated string with no truncation. For this dataset, the difference is negligible (product lists are short), but for very large aggregations, the PostgreSQL version may return longer strings.

### 2. Identity ON NULL Handling

Oracle's `GENERATED BY DEFAULT ON NULL AS IDENTITY` generates a sequence value when either no value or an explicit `NULL` is supplied. PostgreSQL's `GENERATED BY DEFAULT AS IDENTITY` only auto-generates when the column is **omitted** from the INSERT; an explicit `NULL` will raise an error.

### 3. Timestamp Precision

Oracle `TIMESTAMP` defaults to 6 fractional digits. PostgreSQL `TIMESTAMP` also defaults to microsecond (6-digit) precision. No difference expected, but edge-case rounding behavior may vary.

### 4. NUMERIC Rounding

`ROUND(AVG(...), 2)` in the `product_reviews` view: Oracle and PostgreSQL may produce different results for exact midpoint tie-breaking on `NUMERIC` types. In practice, this is unlikely to affect results for this dataset.

### 5. GROUPING SETS with Expressions

The `store_orders` view uses `COALESCE(s.web_address, s.physical_address)` as a grouping expression inside `GROUPING SETS`. Both Oracle and PostgreSQL support this, but PostgreSQL evaluates the expression within each grouping set independently. Behavior is equivalent for this schema.

### 6. JSON Storage

Oracle stores product details as `BLOB` with a `CHECK (IS JSON)` constraint. PostgreSQL uses the native `JSONB` type, which:
- Validates JSON on insert (no separate CHECK needed)
- Stores data in a decomposed binary format (faster queries)
- Does **not** preserve key order or duplicate keys
- Normalizes whitespace

The `product_reviews` view has been rewritten to use `jsonb_array_elements()` instead of `JSON_TABLE()`.

## Validation Suite

The `co_validate.sql` script runs 6 phases of testing:

| Phase | Description | Tests |
|-------|-------------|-------|
| 1 | Schema Validation | Tables, views, data types exist and are correct |
| 2 | Data Validation | Row counts match expected values, spot-check records |
| 3 | Constraint Validation | PK, FK, UNIQUE, CHECK constraints exist |
| 4 | Functional Validation | All 4 views return correct, non-empty results |
| 5 | Identity Validation | Sequences set correctly, auto-increment works |
| 6 | Behavioral Equivalence | Negative tests for constraint violations, JSONB validation |

## Manual Intervention Required

- **None** for a standard installation. The `co_install.sql` script is fully automated.
- If installing into an existing schema that contains other objects, review the `DROP SCHEMA IF EXISTS ... CASCADE` in the install script to ensure it won't remove unrelated objects.
- If using a non-default schema name, pass `-v co_schema=name` to all scripts.
