# Customer Orders Schema — PostgreSQL 17 Port

## Overview

This directory contains the PostgreSQL 17 port of the Oracle 19c **Customer Orders (CO)** sample schema. The migration preserves schema structure, data integrity, constraints, relationships, view semantics, and behavioral equivalence.

## Quick Start

### Prerequisites

- PostgreSQL 17 or higher
- `psql` command-line client
- A database where you have `CREATE SCHEMA` privileges

### Installation

```bash
cd customer_orders/postgresql

# Install with default schema name 'co'
psql -d <database> -f co_install.sql

# Or specify a custom schema name
psql -d <database> -v co_schema=my_co -f co_install.sql
```

### Uninstallation

```bash
# Remove the schema and all objects
psql -d <database> -f co_uninstall.sql

# Or with a custom schema name
psql -d <database> -v co_schema=my_co -f co_uninstall.sql
```

### Validation

```bash
# Run after installation to verify correctness
psql -d <database> -f co_validate.sql
```

## Key Conversion Rules Applied

### Data Type Mappings

| Oracle | PostgreSQL | Notes |
|--------|-----------|-------|
| `VARCHAR2(n CHAR)` | `VARCHAR(n)` | Character semantics preserved by default in PostgreSQL |
| `NUMBER(p,s)` | `NUMERIC(p,s)` | Exact numeric equivalence |
| `INTEGER` | `INTEGER` | Direct mapping |
| `BLOB` (JSON data) | `JSONB` | `products.product_details` — enables native JSON querying |
| `BLOB` (binary data) | `BYTEA` | `stores.logo`, `products.product_image` |
| `DATE` (with time) | `TIMESTAMP` | Oracle `DATE` includes time; PostgreSQL `DATE` does not |
| `TIMESTAMP` | `TIMESTAMP` | Direct mapping |

### Identity Column Handling

| Oracle | PostgreSQL |
|--------|-----------|
| `GENERATED BY DEFAULT ON NULL AS IDENTITY` | `GENERATED BY DEFAULT AS IDENTITY` |
| `ALTER TABLE ... MODIFY ... START WITH LIMIT VALUE` | `SELECT setval(pg_get_serial_sequence(...), MAX(id))` |

**Behavioral difference:** Oracle's `ON NULL` clause means the identity value is generated when `NULL` is explicitly inserted. PostgreSQL's `GENERATED BY DEFAULT` only generates a value when the column is omitted from the INSERT. Explicit `NULL` inserts will fail in PostgreSQL with a NOT NULL violation (since the PK constraint is present). This matches the practical usage pattern where identity columns are either omitted or given explicit values.

### Function Translations

| Oracle | PostgreSQL | Notes |
|--------|-----------|-------|
| `LISTAGG(col, sep) WITHIN GROUP (ORDER BY ...)` | `STRING_AGG(col, sep ORDER BY ...)` | `ON OVERFLOW TRUNCATE` has no PostgreSQL equivalent; full string returned |
| `GROUPING_ID(a, b)` | `CASE WHEN GROUPING(a)=0 AND GROUPING(b)=1 THEN ...` | Explicit CASE with GROUPING() calls |
| `JSON_TABLE(col, path COLUMNS(...))` | `LATERAL jsonb_array_elements(col -> key)` | JSONB operators with LATERAL join |
| `NVL(a, b)` | `COALESCE(a, b)` | Standard SQL equivalent |
| `UTL_RAW.CAST_TO_RAW('json_string')` | `'json_string'::jsonb` | Direct JSONB cast |
| `TO_TIMESTAMP('...', 'DD-MON-YYYY HH24.MI.SS.FF')` | `TIMESTAMP '2021-02-04 13:20:22.245676'` | ISO 8601 timestamp literals |
| `CHECK (col IS JSON)` | *(implicit with JSONB type)* | JSONB type enforces valid JSON on insert |

### Install Script Translations

| Oracle (SQL*Plus) | PostgreSQL (psql) |
|-------------------|-------------------|
| `SPOOL filename` | `\o filename` |
| `@@file.sql` | `\i file.sql` |

| `DROP USER CO CASCADE` | `DROP SCHEMA IF EXISTS co CASCADE` |

## Known Behavioral Differences

### 1. LISTAGG vs STRING_AGG

Oracle's `LISTAGG ... ON OVERFLOW TRUNCATE '...' WITH COUNT` truncates the aggregated string if it exceeds the maximum VARCHAR2 length and appends a count of omitted values. PostgreSQL's `STRING_AGG` has no overflow handling — it returns the full concatenated string. In practice, with `TEXT` type (no length limit), this difference is a benefit rather than a limitation.

### 2. GROUPING_ID vs GROUPING

Oracle provides `GROUPING_ID(a, b)` as a convenience function that returns a bitmask. PostgreSQL supports the standard `GROUPING()` function but not `GROUPING_ID()`. The view uses a `CASE` expression combining individual `GROUPING()` calls for equivalent behavior.

### 3. JSON Storage

Oracle stores JSON as `BLOB` with an `IS JSON` check constraint. PostgreSQL stores it natively as `JSONB` (binary JSON), which:
- Validates JSON structure on insert (no separate CHECK needed)
- Provides native indexing and query operators (`->`, `->>`, `?`, `@>`)
- May reorder keys and remove duplicate keys (JSONB normalization)

### 4. Timestamp Precision

Oracle `TIMESTAMP` defaults to 6 fractional digits. PostgreSQL `TIMESTAMP` supports up to 6 fractional digits (microseconds). The source data uses up to 9 fractional digits (nanoseconds); values are truncated to 6 digits in the conversion. This matches PostgreSQL's maximum precision.

### 5. Identity Columns

Oracle's `GENERATED BY DEFAULT ON NULL` generates a sequence value if `NULL` is explicitly passed. PostgreSQL's `GENERATED BY DEFAULT` only generates a value when the column is omitted. Since all identity columns also have PRIMARY KEY constraints (NOT NULL), explicit `NULL` inserts would fail anyway, making this difference academic for this schema.

### 6. Schema vs User

Oracle creates a `CO` user (user = schema). PostgreSQL uses `CREATE SCHEMA co` within an existing database. The install script sets `search_path` so all objects are created in the `co` schema without requiring table-qualified names.

## Validation Test Suite

The `co_validate.sql` script runs a comprehensive 6-phase test suite:

| Phase | Description | Tests |
|-------|-------------|-------|
| 1 | **Schema Validation** | Tables, views, data types, indexes exist |
| 2 | **Data Validation** | Row counts, spot-check records, timestamps, JSONB |
| 3 | **Constraint Validation** | PK, FK, UNIQUE, CHECK + negative tests |
| 4 | **Functional Validation** | All 4 views return correct results |
| 5 | **Identity Validation** | Auto-increment works after data load |
| 6 | **Behavioral Equivalence** | Aggregations, joins, filters, JSONB queries |

## Manual Intervention

No manual intervention is required for a standard installation. The install script is fully automated and idempotent (it drops and recreates the schema if it already exists).

If you need to install into a specific schema name, use the `-v co_schema=name` parameter.
